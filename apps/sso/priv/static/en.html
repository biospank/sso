    Introduction

    Description

    SSO is a centralized authentication system for Internet and mobile networks. It delivers services (endpoints) in REST mode for registration, authentication, and mail notifications.

    Registered accounts that want to use the authentication system have the following functions available.

    • Account session
    • User registration (double opt-in)
    • User authentication
    • Reset user password (double opt-in)
    • Change user password
    • User details
    • Update user profile

    Session

    Description

    SSO requires access keys for the use of APIs. Contact the Dardy provider to request access keys.

    Example

    
# cURL

curl -i
  -X POST
  -d '{"account": {
        "access_key": "<access_key>",
        "secret_key": "<secret_key>"
      }
    }'
  -H "Accept: application/vnd.dardy.sso.v1+json"
  -H "Content-Type: application/json"
  https://dardy-sso-test.herokuapp.com/sso/session

    # Ruby

require 'httparty'

HTTParty.post('https://dardy-sso-test.herokuapp.com/sso/session',
  body: '{"account": {
            "access_key": "<access_key>",
            "secret_key": "<secret_key>"
           }
         }',
  headers: {
    'Accept' => 'application/vnd.dardy.sso.v1+json',
    'Content-Type' => 'application/json'
  }
)

    // jQuery

$.ajax({
  type: "POST",
  url: "https://dardy-sso-test.herokuapp.com/sso/session",
  data: '{' +
    '"account": {' +
      '"access_key": "<access_key>",' +
      '"secret_key": "<secret_key>"' +
    '}' +
  '}',
  headers: {
    "Accept": "application/vnd.dardy.sso.v1+json",
    "Content-Type": "application/json"
  }
});

    Return a JSON structure

    {
  "account":
  {
    "id":2,
    "app_name": "Test",
    "active": true,
    "access_key": "<access_key>"
  }
}

    HTTP request

    POST https://dardy-sso-test.herokuapp.com/sso/session

    Headers HTTP

    Accept: application/vnd.dardy.sso.v1+json

    Replace <access_key> and <secret_key> with the access keys provided by Dardy.

    HTTP reply headers

    authorization: Dardy <jwt>

    <jwt> is the token required for calls to the other endpoints. x-expires:

    x-expires: <timestamp>

    <timestamp> indicates the session expiry date (1 day)

    HTTP reply codes

    Code Description
    201 Created – The request was successful
    404 Not Found – Account not found
    401 Unauthorized – Account not authorized
    422 Unprocessable Entity – Validation error
    423 Locked – Account temporarily disabled

    User Registration

    Description

    Two forms of registration are available:

    1. Registration with activation
    2. Preauthorized registration

    Registration with activation has two stages:

    • Registration
    • Activation

    Preauthorized registration has one stage:

    • Registration

    Registration with activation

    Description

    This creates an unenabled user whose status is unverified. An e-mail is sent to the address specified by the user in the registration procedure and an e-mail notification is sent to the account address.

    There are two templates for the registration e-mail:

    1. With an activation code (mobile network)
      • Excluding the callback_url from the HTTP message body
    2. With a link to the activation page (Internet)
      • Including the callback_url in the HTTP message body
      • The activation code is appended to the callback_url

    Privacy

    Privacy consent is obligatory and is conceded both to the app/site on which the user registers and the SSO service.

    The obligatory consent items are:

    1. privacy_consent: true
    2. sso_privacy_consent: true

    If registration is completed correctly, the privacy information provided to the app/site is added to the field:

    app_consents: [...]

    Consult this field to check if a user has given privacy consent for a particular app/site.

    Preauthorized registration

    Description

    This creates an active user whose status is verified. A courtesy e-mail is sent to the address specified by the user.

    Privacy

    Privacy consent is obligatory and is conceded both to the app/site on which the user registers and the SSO service.

    The obligatory consent items are:

    1. privacy_consent: true
    2. sso_privacy_consent: true

    If registration is completed correctly, the privacy information provided to the app/site is added to the field:

    app_consents: [...]

    Consult this field to check if a user has given privacy consent for a particular app/site.

    Description

    Password reset includes two stages:

    • Password reset request
    • Reset password

    Password reset request

    Description

    An e-mail is sent to the user who requested a password reset.

    There are two templates for the password reset e-mail:

    1. With a password reset code (mobile network)
      • Excluding the callback_url from the HTTP message body

    An e-mail is sent to the new address of the user who requested to change e-mail address.

    There are two e-mail templates for the change e-mail request:

    1. With a change e-mail code (mobile network)
      • Excluding the callback_url from the HTTP message body
    2. With a link to the change e-mail page (Internet)
      • Including the callback_url in the HTTP message body
      • The change e-mail code is appended to the callback_url

    Example

curl
  -X POST
  -d '{"user": {

        "callback_url": "https//mio.sito.com/cambio/password (opzionale)"
      }
    }'

  -H "Content-Type: application/json"
  -H "Authorization: Dardy <jwt>"

    Update User Profile

    Description

    This updates the user profile with the data submitted in the body of the HTTP message.

    Privacy

    Privacy consent is obligatory and must be submitted by the app/site that registers the user. Consent can be given after registration only if the consent comes from an app/site other than the registering app/site.

    The consent to be submitted is:

    1. privacy_consent: true

    If profile updating is completed correctly, the privacy information provided to the app/site is added to the field:

    app_consents: [...]
